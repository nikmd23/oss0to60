That learning starts with Bruce Wayne Tuckman.

Tuckman's Stages

  1. Forming
  2. Storming
  3. Norming
  4. Performing

Bruce Wayne Tuckman is a researcher in the theory of group dynamics and Professor Emeritus at Ohio State University

In 1965 he published a theory called "Tuckman's Stages"

Tuckman proposed that there are four necessary and inevitable phases in order for a team to grow, to face up to challenges, to tackle problems, to find solutions, to plan work, and to deliver results

Those phases are:

  • Forming - the period in which a team comes together
  • Storming - the phase when a team runs into inevitable challenges
  • Norming - the phase when a team understands the plan and can execute on it
  • Performing - the phase when individuals on the can begin to anticipate each other

OSS is special because the team is constantly churning - so there are always people in each of these stages.

Let's explore each of these in more detail...

Forming

The first phase, is forming

While Forming, serious issues are avoided, and people focus on being busy with meta work, such as organizing the team, deciding who does what, where the code lives, etc.

I've pulled together three points to consider while forming, and perhaps the most important one for an OSS project is the thing that makes a project OSS in the first place: it's license

Select a License

tl;dr Legal to the rescue!

Selecting a license can be daunting. (Probably best not to create your own!) But it is important to set the tone for how the project can be distributed ASAP.

Contributors and users may be attracted (or repeled) to a project based on it's license. So choose carefully based on your target audience

Story Time For the 1st release of Glimpse we chose a Creative Commons license - simply because we didn't know any better and they had a nice license landing page.

Unfortunately, as Phil Haack pointed out to us, CC licenses are better suited for content, and he suggested we take a look at the Apache 2 license.

To figure out which license is right for you, be sure to select one from OpenSouce.org since their licenses all comply with the Open Source Definition, which defines 10 tenants of a truly open source license.

tldrlegal.com really comes in handy for gaining a quick overview of a licenses and for comparing a license

Demo Apache 2 on tldrlegal

Contributors License Agreements are another type of license you may want to consider for your project.

CLA's are agreements that contributors to a project sign turning over their copy write to the project

If you are asked to sign one, do not be alarmed - they are fairly common

Glimpse has one, also at the suggestion of Phil Haack and it has served as well since Redgate asked about copy write when they sponsored the project

CLA's are less organized/categorized than standard licenses, but typically come in flavors for individuals and corporations

Try to use a CLA template like the one available from the Apache Foundation to simplify the process for contributors.

Project Setup

  • Website
  • Mission Statement
  • Contributor Guidelines
  • Screenshot/Demo
  • Consider Forkability

OSS projects need a website. Websites provide a lot more info than just what your repository or package feed can cover.

Story time We learned this lesson the hard way from Scott Hanselman, who demanded that we have a website for Glimpse before he would blog about it. I stayed up until 4:00 AM cranking out the 1st Glimpse website on Tumblr.

The website is often the first impression that a user has of the project, so take some care to get a decent template. It can be simple, but should be pleasant to use.

As a user should feel free provide feedback on the website as well. Often the website is open source itself so it can be contributed to.

Mission statement/readme.md that expresses the goals of the project and what it will/won't do

scriptcs has a nice simple mission statement: scriptcs makes it easy to write and execute C# with a simple text editor. It's concise and to the point.

Contributor guidelines should be available to to inform developers how to get the source, build the source and submit pull requests. Be sure to cover any conventions or standards used within the project.

If your looking to contribute to a project - be sure to look for this!

Contributor guidelines are so important to open source that GitHub has built in support for them via the contributing.md file

Demo/screenshots have always been a recommended best practice for commercial projects, and the same holds true for OSS

Story time The 4:00 AM YouTube video of Glimpse had 50K views and dramatically helped people grok the concept behind glimpse

Forkability, a term coined by Karl Fogel of the SVN project, is the ease of which a developer could fork the project. Forkability also makes it easy for them to grab the code, build it and get started.

Some companies wearing the fact that they can get a new hire to deploy to production on their first day on the job as a badge of honor. How long does it take to get a new contributor up and running in a OSS project is a measure of its forkability.

To make a codebase forkable:

  • Keep as many dependencies in source control/package restore as possible
  • Single command build process
  • Built in samples
  • Developer documentation
  • Forkability helps to ensure distributed leadership

Because a fork could produce a pull request or a competitor - forkability keeps maintainers honest and working for the good of the project

Honesty and transparency are also fundamental tenants of governance models

Selecting a Governance Model

Meritocratic Model @ OSS Watch

Contributors should get acquainted with the governance model and try to act accordingly

Defining a governance models for a project is important to help orient users with the way to interact with given project's community

There are many different governance models, mostly ad hoc, and two particularly famous ones: benevolent dictator and meritocratic method

Benevolent dictator is the model that Linus Torvolds and the Linux team leverage. I personally am not a huge fan of this model, but it clearly does work

For today I am going to focus on the meritocratic method, because that's the one I have experience with

I though I knew what the meritocratic method was since so many project claim that they work in tht model - but I was shocked to find that there is well written governance documents by OSS-Watch which explains all the ins and outs in full details

I'm going to quickly recap by covering the Roles and decision making process

Users are people who actually need the project. They are the most important members of the community and without them the project would have no purpose. Obviously anyone can be a user; there are no special requirements. Users provide feedback, word of mouth evangelism, moral and sometimes financial support.

Contributors are community members who contribute in concrete ways to the project. Anyone can become a contributor, and contributions can take many forms such as support, reporting bugs, identifying requirements, providing graphics and web design, writing documentation, fixing bugs, adding features

Committers are community members who have shown that they are committed to the continued development of the project through ongoing engagement with the community. Committership allows contributors to more easily carry on with their project related activities by giving them direct access to the project's resources. That is, they can make changes directly to project outputs, without having to submit changes via patches. A committer seeks approval after the contribution is made, rather than before. They must be nominated.

Reviewers consists of those individuals identified as "project owners". The reviewers have additional responsibilities over and above those of a committer. These responsibilities ensure the smooth running of the project. Reviewers are expected to review code contributions, participate in strategic planning, approve changes to the governance model and manage the copyrights within the project outputs. They must be nominated.

A meritocracy is not a democracy; that is, not everyone has a binding vote. Only those who have earned it through positive contribution to the project have a binding vote.

There are a full set of rules for which decisions require which type of vote when a concensus can't be naturally achieved - check them out on OSS Watch

Once a project is setup and relatively well formed, it will begin to enter into the 2nd of Tuckman's stages.

Storming

OSS is built by humans, and as humans we will have conflicts. Ideas begin to compete with each other. Personalities (and egos!) begin to battle. Architectural challenges and bugs begin to crop up

The storming stage is necessary for the growth of the project. It can be contentious, unpleasant and painful, but projects that make it past this stage come out stronger.

Communication

Incoming

Outgoing

Internal

It should be no surprise to see the communications is the first challenge to discuss within OSS projects.

Good and proper communication is the most important part of managing an OSS project. Lack of good communication can cause a project to break down and lose contributors. Communications can be broken into three channels.

Incoming from Forums, Mailing Lists, Issue Trackers, etc. Receiving and processing feedback and viewpoints for the purpose of improvement. An example of this could include surveys to determine how well a part of your community is working.

Outgoing to blogs and social media. Sharing news, stories, and achievements from the community with the rest of the world. An example of this could be showing off something your community has created.

Internal with HipChat, Trello and Skype should be used sparingly. Internal discussions and meetings in the community to discuss objectives, goals, conflict, and other issues. An example of this could include meetings that are designed to decide how your community will work together toward its goals.

In general, it's best to avoid private/internal communications - save your breath and keystrokes for more productive purposes

Once you have settled on where to communicate it's important to understand how to communicate.

Nom de plume

the only thing anyone knows about you on the Internet comes from what you write, or what others write about you. You are your nom de plume, you are your pen - or keyboard in this case.

You may be brilliant, perceptive, and charismatic in person—but if your emails are rambling and unstructured, people will assume that's the real you. You could be a dog - and know one will know as long as your eloquent, thoughtful and well spoken

So here's just a few quick tips for users, contributors reviewers and the like. They shouldn't need repeating, but in my experience, they do

Your avatar and handle are your face - try to be consistent

User proper grammar, spelling and punctuation

Consider subject/title lines carefully

Use a diagram if it will help! (AsciiFlow & WebSequenceDiagrams)

Provide context! Use a link to another thread or an issue . Make it easy for the reader

Watch your tone

Even well meaning, properly placed communications do no good if they aren't actually helping the project to progress forward. It is important to keep progress in mind while communicating, and one common enemy of progress it a technique called bikeshedding.

Avoid Bikeshedding

Fwd bikeshed.com

In his 1957 book Parkinson's Law, and Other Studies in Administration, Cyril Northcote Parkinson introduced "Parkinson's law of triviality" which states:

the amount of noise generated by a change is inversely proportional to the complexity of the change

Story TimeParkinson dramatizes this law with the example of a committee's deliberations on building an atomic power plant, contrasting it to deliberations on building a bicycle shed.

The reason this happens is because people want to prove that they are involved and to "put their fingerprints on something" so that one day they can walk by and say "I built that".

Unfortunately whats really happening is that the community is toiling in minor issues while there are larger, higher priority problems to tackle

In 1999 Poul-Henning Kamp applied Parkinson's law to the OSS FreeBSD project in an email to the projects mailing list. He said:

"Why Should I Care What Color the Bikeshed Is? The really, really short answer is that you should not. The somewhat longer answer is that just because you are capable of building a bikeshed does not mean you should stop others from building one just because you do not like the color they plan to paint it. This is a metaphor indicating that you need not argue about every little feature just because you know enough to do so. Some people have commented that the amount of noise generated by a change is inversely proportional to the complexity of the change."

Kamp's email is available at bikeshed.com - send it to the member of your community and make them aware of the concept and your desire to avoid it. Encourage them to point out the behavior when they see it as well.

Besides communication challenges OSS projects face many of the same challenges as marriages and other relationships. We all know that finances can put huge stress on a marriage - the same goes for OSS projects.

Dealing w/ $$$

Finances can be the biggest cause of strain on any relationship, including the relationship between an open source team.

We were lucky with Glimpse because we were tested with money very early on.

Story TimeAt Mix we were offered 10K to close and sell the source, which forced us to evaluate why we really did Glimpse.

We decided to back pay our bills first, and spend anything left over on thank you swag for contributors

On the other side of the spectrum, some projects can't stay afloat because their maintainers can't afford to pay any bills that may be associated with the project. If you enjoy using an OSS product, and don't have the time to get involved in the effort, consider donating $$.

Check to make sure they have a plan for the cash first, to avoid the situation that Jeff Atwood had with his 5K donation to screwturn wiki

And finally, for the last challenge I've personalyy experienced, I wanted to talk about OSS/Life Balance

OSS/Life Balance

Congrats - you have a new baby boy for a project. You're so proud, people are using it and it's going to be awesome.

People use this analogy of the "parents" of a project. And children can be a real joy and sense of pride in a parents life.

But along with that pride and joy comes dirty diapers and a lot of work.

With OSS there is a lot of pressure to fix bugs, support users, grow community, make it better, etc

What makes it worse is that the number of issues in a project's bug tracker grows in proportion to the number of people using the software. Therefore, even as you fix bugs and ship an increasingly robust program, you should still expect the number of open issues to grow essentially without bound.

I wrestled mightily with this. Working all weekend to ensure I didn't miss a "deadline". Making sure there was a plan to fix everybug. Answering every support question that came in. Crying myself to sleep.

I don't have a clever well thought out approach or tip here. I'm just being honest and wanted to leave you with encouragement.

I just want to remind you that OSS isn't life. There are more important things out there. Don't take yourself so seriously. Remember why you started OSS in the first place.

The good news is, if you can make it out of the storming phase, things really start to get fun during Tuckman's 3rd stage: Norming

Norming

When teams are norming the have a clear goal and a mutual agreement for the plan of execution.

Once the plan of execution is in place, it's fairly normal to begin to see a shift from development work to community management work.

Community Management

Contributors and users should try to not pull the team off their goals too much. Be mindful of support requests. Give back the help you receive.

Once the project has gotten setup and gone through its first round of problems, the software will be the easy part, getting people aligned and "herding cats" is the challenge

First, you'll want to treat every user as a potential volunteer. String them along the process building an on ramp for them if possible.

To manage the community, you want to keep in mind two things: focus and attention

Focus the community to be working on the task at hand. Defining milestones helps with this.

Bring attention to the focused efforts of the community. We like to do this with town hall meetings on Google Hangouts where we ask contributors to show off their recent work

Meeting in person, or less so on Skype, also really strengthens a community. There is something about breaking bread together and sharing a few drinks that really helps the bonding process

Story Time We met a user in Brussels recently - he had never contributed to an OSS project before. But after a few hours and lots of laughs we had built up a repore that continues virtually over the internet now. He's quickly becoming one of our top contributors within a short time span.

Another nice thing about the norming phase is that once a project is decently established, it can usually begin to enjoy some of the perks that are provided by vendors in the industry

Once the community is humming along and tooling is in place, you'll find that your project team can really start performing

Performing

When a team is performing, the individuals function as a unit without the need for heavy management oversight.

Now, there are a few things that can be done to reduce management oversight and to begin performing earlier. The first of which is to automate everything.

Automate Everything

Contributors should pay attention to these mechanism - if they aren't forced upon them

Managing your project and the project's community takes time. Lot's of time. Simplify your life and let the robots help you

This is where perks really start to shine: enforce build and test procedures with a CI server

Enforce style and naming conventions automatically instead of with a document via tools like StyleCop

Auto test pull requests and leverage GitHub's commit status API - built into TeamCity and TravisCI

Story Time Story of how user's automatically fix pull requests now without our involvement

Automation can also be as simple as opening up your documentation contribution process. We leverage Glimpse's GitHub wiki repo as a subrepo on the website to allow users to directly contribute docs without the need for a pull request. Documentation contributions have tripled since we've done this.

Automation is great, but thinking about contributions and new contributors can also really increase performance.

Create Opportunity

Jump In Issues

Story Time Story of users saying that they didn't have enough breath to add a new feature or enough depth to fix a bug, which is how this was inspired

Help new contributors by setting up Jump In tags on issues that typically:

  • Take no longer than a few nights worth of work to implement.
  • Be relatively standalone, not requiring tight integration with other in development backlog items.
  • Be well described with pointers to help the implementer.
  • They are not:
  • Items that the “core team” does not want to implement themselves.
  • Items that are specifically set aside for junior/novice developers.
  • Used as a hazing activity for “rookies”.
  • Designed to be fizzbuzz tests.

If you are new to a project and looking for ways to contribute, ask if they have any Jump In tasks for you

Thank contributors as loudly as possible

In the same vein to make things easier for new contributors, thanks those who have completed work as loudly as possible
